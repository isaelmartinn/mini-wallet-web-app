# Implementation Plan 0: Infrastructure and Core Setup

## Context and Objective
Establish the project's technical foundation following the architecture decisions defined in `DECISIONS.md`. The goal is to configure the development environment with code quality tools, testing frameworks, and the base folder structure to support Hexagonal Architecture and DDD.

## Architecture and Design
- **Folder Structure**: Creation of `/src/contexts`, `/src/shared`, `/app`, `/e2e`.
- **Code Quality**: ESLint (with boundaries), Prettier, Husky.
- **Testing**: Vitest for unit/integration tests and Playwright for E2E.
- **UI**: Tailwind CSS and shadcn/ui.

## General Implementation Rules (Applicable to all plans)
- **Language**: Code, variable names, functions, classes, internal documentation and comments **ALWAYS** in English.
- **UI**: Text visible to the end user **ALWAYS** in Spanish.
- **Components**: Do not create custom components if a `shadcn/ui` alternative exists. **ALWAYS** check the library before implementing. Only create components if the logic is very business-specific or does not exist in the library.

## Implementation Steps

### 1. Code Quality, Formatting, and Editor Configuration
- Create `.editorconfig` file synchronized with Prettier and ESLint rules.
- Configure `ESLint` with the rules defined in `DECISIONS.md`.
- Implement `eslint-plugin-boundaries` to define boundaries between layers (domain, application, infrastructure) and domains (auth, wallet, transactions).
- Configure `Prettier` for consistent formatting.
- Configure `Husky` and `lint-staged` to run linting and fast tests on pre-commit.

### 2. Boilerplate Cleanup
- Remove files generated by Next.js CLI that won't be used (initial `app/page.tsx`, `public/next.svg`, `public/vercel.svg`, etc.).
- Clean `app/globals.css` to leave only Tailwind directives and minimal base styles.

### 3. TypeScript and Alias Configuration
- Configure `tsconfig.json` with defined path aliases: `#auth/*`, `#wallet/*`, `#transactions/*`, `#shared/*`.
- Enable TypeScript strict mode.

### 4. Testing Framework Setup
- Install and configure `Vitest` with `React Testing Library`.
- Configure global test environment and base mocks.
- Install and configure `Playwright` for E2E testing in the `/e2e` folder.

### 5. Shared Structure and Core UI
- Create initial structure in `/src/shared`: `domain`, `infrastructure`, `ui`.
- Configure `Tailwind CSS` and `PostCSS`.
- Initialize `shadcn/ui` and add necessary base components (Button, Input, Card).
- Implement `DomainError` base class in `/src/shared/domain/errors`.

### 6. Base Layout and Next.js Routes
- Configure `app/layout.tsx` with fonts and global styles.
- Create route groups `(auth)` and `(dashboard)` in `/app`.

## Acceptance Criteria
- `npm run lint` passes without errors and blocks boundary violations.
- `npm run test` runs unit tests correctly.
- TS aliases work correctly in imports.
- Husky blocks commits if lint or related tests fail.

## Test Plan
- **Unit**: Verify that `DomainError` class instantiates correctly.
- **Integration**: Verify that a basic shadcn component renders with Tailwind.
- **E2E**: Smoke test verifying the app starts and shows the root page.

## Risks and Mitigations
- **Risk**: Initial complexity in boundary configuration.
- **Mitigation**: Define rules incrementally and validate with simple import examples.

## Recommended Tools and Guides
- `eslint-plugin-boundaries` documentation.
- `shadcn/ui` installation guide.
- `vitest` configuration docs.
